Cómo optimizamos el dashboard de DevFlow resolviendo un cuello de botella N+1 en PostgreSQL

Una crónica de cómo pasar de 50 consultas de base de datos a un solo JOIN eficiente, reduciendo el tiempo de carga de 8s a 200ms.

T
Escrito por Tomás Desarrollador Full Stack

1. Contexto

El desafío ocurrió dentro de DevFlow, una plataforma web interna de analíticas desarrollada con un backend en Node.js (Express), una base de datos PostgreSQL, y utilizando Sequelize como ORM. Esta plataforma permite a los líderes de equipo visualizar métricas críticas de productividad en tiempo real, incluyendo los commits recientes y las tareas completadas por cada desarrollador del equipo.

2. El Problema: El cuello de botella invisible

A medida que el equipo de desarrollo creció a más de 50 desarrolladores, la pantalla principal del dashboard comenzó a degradarse severamente. El tiempo de respuesta de la API (TTFB) superaba los 8 segundos, frustrando a los usuarios.

Al inspeccionar los logs de PostgreSQL en nuestro entorno de pruebas, descubrimos una ráfaga inmensa de consultas redundantes para una sola petición de página. El ORM estaba haciendo lo siguiente:

  1. Una consulta inicial para obtener a todos los usuarios del equipo.
  2. N consultas adicionales (una por cada usuario retornado) para buscar sus commits y tareas correspondientes.
Ejemplo de Consulta Ineficiente (N+1) SQL / Logs 
-- 1. Obtener usuarios (Consulta Inicial)
SELECT "id", "name", "email" FROM "Users" WHERE "teamId" = 1;

-- 2. Consultas N (una por cada usuario devuelto, ej: user_id=101, 102...)
SELECT "id", "hash", "message" FROM "Commits" WHERE "userId" = 101;
SELECT "id", "hash", "message" FROM "Commits" WHERE "userId" = 102;
-- ... [Repetido 50 veces] ...

Este es el clásico problema de consultas N+1. En un entorno de desarrollo local con 3 usuarios de prueba el retraso era de milisegundos e imperceptible, pero con un volumen real en producción el pool de conexiones se saturaba completamente.

3. Acciones Tomadas y Post-Mortem

1

Diagnóstico & EXPLAIN ANALYZE

Corrimos un análisis directo en la base de datos usando EXPLAIN ANALYZE para verificar el tiempo de escaneo secuencial y el costo de planificación en las tablas de Commits y Tasks.

2

Refactorización a Eager Loading

Reescribimos la lógica del controlador para pasar de la carga diferida (lazy loading) a la carga ansiosa (eager loading) en Sequelize mediante la cláusula include. Esto forzó a Sequelize a generar una única consulta consolidada con un LEFT OUTER JOIN a nivel de base de datos.

Refactorización de Sequelize Javascript 
// ANTES (Causaba N+1 al iterar en el frontend o serializador)
const users = await User.findAll({ where: { teamId } });

// DESPUÉS (Una sola consulta optimizada con JOIN)
const users = await User.findAll({
  where: { teamId },
  include: [
    { model: Commit, as: 'commits' },
    { model: Task, as: 'tasks' }
  ]
});
3

Indexación Estratégica

Creamos índices compuestos en las columnas que actúan como claves foráneas (userId, teamId) y de ordenación (createdAt) para optimizar la velocidad del JOIN.

4

Post-Mortem Constructivo

Realizamos una sesión con el equipo de ingeniería para documentar el incidente. Concluimos que las pruebas de regresión se hacían con bases de datos vacías o con semillas minúsculas. Establecimos como regla obligatoria usar semillas de al menos 100 usuarios en pruebas de desarrollo y configurar alertas de rendimiento para tiempos de respuesta superiores a 500ms en el entorno de staging.

4. Aprendizajes y Conclusiones

1. Entender la Abstracción del ORM

Los ORM facilitan enormemente el desarrollo rápido, pero ocultan el comportamiento de la base de datos subyacente. Siempre debemos revisar y auditar las consultas SQL generadas en la consola durante la fase de desarrollo.

2. Semillas de Prueba Realistas

Las pruebas funcionales no son suficientes. Si los datos de prueba no replican la escala de la producción, los problemas de rendimiento y arquitectura se harán visibles recién frente a los usuarios finales.

3. Observabilidad Primero

Tener métricas claras y logs estructurados de base de datos nos permite diagnosticar problemas en minutos en lugar de horas.

5. Reflexión sobre Feedback Radicalmente Sincero

Durante la revisión de código (Pull Request), mi primer intento de solucionar la lentitud consistió en proponer un sistema de caché con Redis para guardar el dashboard formateado.

Mi compañero de equipo me dio un feedback extremadamente directo y sincero:

"Colocar una base de datos en caché para tapar una consulta N+1 es una mala práctica. Estás agregando complejidad operativa (Redis), latencia de invalidación y costos adicionales de infraestructura para ocultar un problema que se soluciona con una sola línea de código en Sequelize. Resolvamos la raíz del problema en la base de datos primero."

Inicialmente, sentí frustración porque la caché de Redis ya la tenía programada y funcionando. Sin embargo, al reflexionar bajo el marco del Radical Candor (Feedback Radicalmente Sincero), entendí que su crítica no era personal, sino orientada a la excelencia técnica del producto. Descarté la implementación de Redis, investigué cómo usar Eager Loading eficientemente en Sequelize y resolví el cuello de botella de raíz. Esto ahorró costos de servidor y nos dejó un codebase mucho más limpio y mantenible.
